## 1. Компоненты  
### Ride Service (сервис, рассчитывающий стоимость поездок)
Сервис предоставляет API Gateway'ю (которого вызывает заказчик из мобильного приложения) следующий ФУНКЦИОНАЛ (к теме ФТ):
1. Расчёт стоимости поездки для показа пользователю (скорее всего, пред-показа, до кнопки Заказать)
2. Размещение заказа на поездку ( create() - endpoint )
3. update() - ? возможно, вызывается при отсутствии откликов от водителей при некой попытке приложения найти водителя за несколько больший прайс, чем был изначально
4. accept() - ? для update() - предлагает пользователю поехать за несколько больший прайс, т.к. за указанную ранее цену никто не соглашается за timeout-промежуток

К теме НФТ:
1. Общается со своей Primary DB (в которой есть данные для расчета цены). Например, погодные условия, протяженность, базовые и усложненные коэффициенты (по бизнес-логике)
2. Общается с некими внешними интеграциями (те же данные о погодных условиях) - через 3rd party mapping
3. Связан с Ride matching Service методом получения статуса водителя (вероятно, доступен/уже выполняет длительный заказ)

### Ride matching Service (сервис, предоставляющий водителю заказы, а заказчику — такси)
Вызывается клиентскими мобилками User/Driver через посредника в виде шлюза (безопасность, логи, трейсинг, роутинг и пр.).
1. Через функцию RequestRide() добавляет запрос такси (вызывается User'ом) ИЛИ заказа (вызывается Driver'ом) в очередь Ride request queue.
2. Из очереди сервис вычитывает заявки И:  
   a) Подбирает ближайших водителей через Location DB (Redis geohashing). High-load база данных (100k-1M tps)  
   b) Бронирует водителя на поездку (DriverId) на 10 секунд - timeout/estimate на принятие решения водителем - везти или не везти (брать заказ или не брать) записью в Redis "статусов водителей"(?)  
   c) В случае match'а (успешно подобрались) - push или другая нотификация через NotificationService водителю.

### Location Service — сервис обновления данных о геолокации водителей для оперативного поиска ближайших водителей к заказчику
1. Вызывается через тот же шлюз, что и два сервиса выше. Цель - поддержка актуальности данных о геолокации водителей (прогрев "кэша" (Redis geohashed)). High-load: 6M drivers, 3M active drivers. 600K tps.

### API Gateway (не бизнес-логика компонент)
1. Балансировка нагрузки между разными инстансами сервисов
2. Маршрутизация запросов сервисам (service discovery)
3. Аутентификация = проверка учетных записей пользователей и водителей
4. Rate Limit: во избежание перегрузки или, скорее, DDOS атак

### 3rd party mapping
Для интеграции с внешними (3rd party) системами. Например, для получения актуальных данных о погодных условиях. В зависимости от этого цена на поездку может быть выше. Мэтчинг наших и их контрактов.

### Primary DB
База данных микросервиса (судя по диаграмме, в него ходит только 1 сервис) Ride Service. Может содержать данные для расчета стандартных или повышенных коэффициентов. Например, те же данные о погоде можно класть туда, чтобы  
не дергать интеграцию много раз (HTTP overhead) тем более если она платная.

### Redis (driverId)
Поиск машины для заказчика нужно осуществлять максимально быстро, учитывая конкуренцию на рынке. В этой связи есть смысл рассматривать Redis, работающий в оперативке.

### Location DB (Redis, geohashed)
Местоположение водителей также должно быть всегда максимально актуальным для оперативного (максимально оперативного) поиска водителя для заказчика и оптимизации расходов на топливо.

### NotificationService
Пуши на мобилки, скорее всего. Либо SMS. Некий сервис-шлюз отправки уведомлений в зависимости от типа (SMS, Push).

## 2. Потоки данных  
1.	Пользователь размещает заказ (стоимость была получена getFareEstimate при первом входе и выборе конечной точки маршрута):
      User (mobile) --> API Gateway (rate limit/auth, balance, routin) --> RequestRide() --> Ride matching Service --> бронь водителя к заказу на время размышлений водителя брать/не брать. Далее идет обратная цепочка и 1) водитель принял заказ – юзер это видит, ждет и едет. Либо отменяет позже. 2) водитель не принял – ищем другого до достижения тайм-аута. Также откидывается уведомление в сервис нотификаций.
2.	Водитель «выходит на линию»:
      Driver --> Gateway --> 1) обновляется локация с целью актуализации данных и большей оперативности поиска водителя под заказы в локациях рядом. 2) идет запрос заказа на поездку по аналогии с заказом от заказчика (1 сценарий).
3.	Заказчик заходит в приложение и видит стоимость поездки предварительную благодаря getFareEstimate() с учетом данных по загруженности водителей и т.д (погода, коэффициенты расчета из интеграций 3rd party).
