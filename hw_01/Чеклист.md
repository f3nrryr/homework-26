# Чек-лист для анализа референсной архитектуры (ДЗ 1)

Используйте этот чек-лист, чтобы разобрать выбранную референсную архитектуру
(облачное хранилище, рекламная система, видеостриминг или сервис такси).
Идите сверху вниз — каждый блок опирается на предыдущий.

---

## 0. Контекст системы

- [ ] Сформулировал, **что делает система** в 1-2 предложениях
- [ ] Назвал основных **пользователей / акторов** (кто и зачем обращается к системе)
- [ ] Выписал **ключевые сценарии использования** (3-5 штук) — что система делает чаще всего
- [ ] Отметил **масштаб** (порядок: тысячи / миллионы пользователей, объём данных, RPS) — хотя бы грубо

---

## 1. Компоненты (5-10 штук)

Для каждого компонента:

- [ ] Указал **название** и **тип** (сервис, БД, кэш, очередь, CDN, балансировщик, внешняя интеграция)
- [ ] Описал **зону ответственности** в 1-2 предложениях («за что отвечает»)
- [ ] Проверил, что нет **«god service»** (один компонент делает всё)
- [ ] Проверил, что нет **избыточного дробления** (компоненты без чёткой причины существовать отдельно)
- [ ] Отделил **stateless-компоненты** (можно масштабировать копированием) от **stateful** (хранят состояние)

> Подсказка: начинайте с вопроса «какие компоненты я ожидаю увидеть внутри?»

---

## 2. Потоки данных (3-5 ключевых сценариев)

Для каждого сценария (например: «пользователь загружает файл», «показ рекламы», «заказ такси»):

- [ ] Описал путь запроса **от пользователя до хранилища и обратно**
- [ ] Указал, **какие компоненты** участвуют и **в каком порядке**
- [ ] Отметил, где взаимодействие **синхронное**, а где **асинхронное** (очередь, событие)
- [ ] Нарисовал **схему** потока (стрелки = направление данных)
- [ ] Указал, что происходит при **сбое** на ключевом шаге (retry, fallback, ошибка пользователю)

---

## 3. Проблемы и решения

Для каждого важного компонента / решения:

- [ ] Сформулировал, **какую проблему** он решает («зачем он здесь»)
- [ ] Указал, **что сломалось бы без него**
- [ ] Отметил, какие **trade-offs** приняты (что выиграли, чем пожертвовали)
- [ ] Нашёл хотя бы 1-2 места, где видно **масштабирование** (шардирование, репликация, кэш, CDN)
- [ ] Нашёл хотя бы 1 место, отвечающее за **надёжность** (репликация, очередь как буфер, фоллбэк)

---

## 4. Вопросы к авторам архитектуры (минимум 5)

- [ ] Вопросы **не очевидны** из схемы (не «что такое БД», а «почему выбрали именно эту БД»)
- [ ] Есть вопросы про **выбор технологий и trade-offs**
- [ ] Есть вопросы про **поведение под нагрузкой / при сбоях**
- [ ] Есть вопросы про **узкие места** (что упрётся первым при росте)
- [ ] Каждый вопрос показывает, что вы **поняли архитектуру**, а не просто прочитали схему

---

## Самопроверка перед сдачей

- [ ] Документ в формате Markdown, объём 3-5 страниц
- [ ] Есть схема(ы) — хотя бы для потоков данных
- [ ] Компоненты и ответственности выделены корректно (критерий 1)
- [ ] Ключевые потоки данных описаны со схемами (критерий 2)
- [ ] Вопросы осмысленные, не очевидные из схемы (критерий 3)

> **Статус «Принято»** ставится при выполнении всех трёх критериев оценки ДЗ 1.
