# Порядок сдачи домашних заданий

Документ описывает, как работать с репозиторием при выполнении ДЗ: ветки, коммиты,
Pull Request и приёмку работы преподавателем.

## Кратко

1. На каждое задание — отдельная ветка `hw_NN`.
2. Решение кладётся в `hw_NN/solution/`.
3. По готовности — Pull Request из ветки `hw_NN` в `main`.
4. Преподаватель ревьюит PR; правки вносятся новыми коммитами в ту же ветку.
5. **Merge в `main` — только после статуса «Принято».**

## 1. Подготовка

Перед началом заполните шапку в [README.md](README.md): ФИО и группу.

```bash
git clone <url-репозитория>
cd homework-26
```

Работайте всегда в актуальном `main`:

```bash
git checkout main
git pull
```

## 2. Ветка под задание

На каждое ДЗ заводится отдельная ветка с именем по номеру задания: `hw_01`, `hw_02`,
… `hw_06`.

```bash
git checkout main
git pull
git checkout -b hw_01
```

## 3. Выполнение

- Решение оформляется в `hw_NN/solution/README.md` (Markdown).
- Схемы (PlantUML / C4 / PNG) кладутся в `hw_NN/solution/diagrams/`.
- Файлы `Задание.md`, чек-листы и шаблоны **не изменяются** — это исходные материалы.
- В таблице в [README.md](README.md) обновите статус задания на `🟡 В работе`.

Коммитьте осмысленными порциями:

```bash
git add hw_01/solution
git commit -m "hw_01: анализ компонентов и потоков данных"
git push -u origin hw_01
```

### Договорённости по коммитам

- Сообщение начинается с номера ДЗ: `hw_01: ...`.
- Описывайте суть изменения, а не «правки»/«фикс».
- Не коммитьте временные и бинарные артефакты (экспортные файлы редакторов,
  `*.bak`, кэш IDE). При необходимости добавляйте их в `.gitignore`.

## 4. Pull Request

Когда работа готова, откройте Pull Request из ветки `hw_NN` в `main`:

- Заголовок: `ДЗ N. <название задания>`.
- В описании коротко: что сделано и где (ссылка на `hw_NN/solution/`).
- Убедитесь, что выполнены все пункты «Критерии оценки» из `Задание.md`.

## 5. Ревью и приёмка

- Преподаватель оставляет комментарии в PR.
- Доработки вносятся **новыми коммитами в ту же ветку** `hw_NN` (PR обновляется
  автоматически) — не открывайте новый PR на доработку.
- Когда работа получает статус **«Принято»**:
  - PR вливается в `main` (merge),
  - в таблице [README.md](README.md) статус задания меняется на `✅ Принято`.

## Статусы в README

| Обозначение | Значение |
|-------------|----------|
| ⬜ Не начато | работа ещё не начата |
| 🟡 В работе | есть ветка/PR, работа идёт |
| ✅ Принято | работа принята и влита в `main` |
