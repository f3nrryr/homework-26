# Чек-лист метрик RED / USE (ДЗ 5)

Опора для пункта «Список ключевых метрик (15–20 по RED/USE)». Чек-лист задаёт
**рамку**, по которой вы выводите метрики для своей системы — конкретные метрики
под свои сервисы выбираете сами. Оценивается осмысленность, а не копипаста списка.

---

## Когда RED, когда USE

- **RED** — для **сервисов**, обрабатывающих запросы (API, обработчики). Смотрит со стороны клиента.
- **USE** — для **ресурсов** (CPU, память, диск, сеть, пул соединений, очередь). Смотрит со стороны железа/ресурса.

> Практика: для каждого сервиса — RED, для каждого ресурса под ним — USE. Плюс несколько бизнес-метрик.

---

## RED — на каждый ключевой сервис

- [ ] **Rate** — запросов в секунду (по эндпоинтам / типам операций)
- [ ] **Errors** — доля и количество ошибок (4xx отдельно от 5xx, таймауты)
- [ ] **Duration** — латентность в перцентилях: p50, p95, p99 (не среднее!)

## USE — на каждый ресурс

- [ ] **Utilization** — насколько занят ресурс (CPU %, memory %, IOPS)
- [ ] **Saturation** — очередь/насыщение (load average, длина очереди, lag)
- [ ] **Errors** — ошибки ресурса (OOM, дисковые ошибки, отказы соединений)

Типовые ресурсы для проверки: CPU, RAM, диск (место + IOPS), сеть, **пул соединений к БД**,
**lag репликации**, **глубина очереди брокера**, file descriptors.

## Бизнес / golden signals (не забыть)

- [ ] Метрики, отражающие **ценность для пользователя** (успешные заказы/платежи/бронирования в минуту)
- [ ] Метрики **насыщения бизнес-потока** (отвалившиеся корзины, очередь на обработку)

---

## Вопросы, чтобы не собирать «метрики ради метрик»

1. Если эта метрика выйдет за порог — **что я предприму**? (нет действия → метрика лишняя)
2. Покрыты ли все **критичные сервисы** из ДЗ 2 (RED) и их ресурсы (USE)?
3. Видно ли по метрикам **деградацию до того**, как её заметит пользователь?
4. Различаю ли я **симптом** (рост p99) и **причину** (насыщение пула)?

---

## Шаблон для заполнения

| Сервис / ресурс | Метод | Метрика | Как меряем | Зачем (что покажет) |
|---|---|---|---|---|
| _напр. payment-api_ | RED | p99 latency | гистограмма по эндпоинту | деградация отклика для клиента |
| _напр. PostgreSQL_ | USE | utilization пула соединений | % занятых connections | приближение к лимиту до отказов |
| … | | | | |

> Цель ДЗ — 15–20 метрик. Хороший набор покрывает RED по сервисам + USE по ресурсам + 2–3 бизнес-метрики.
> Слабый набор — список технических метрик «потому что их собирает Prometheus по умолчанию».

---

## Самопроверка

- [ ] Латентность — в перцентилях, не среднее
- [ ] Ошибки разделены по типам (4xx / 5xx / таймауты)
- [ ] Покрыты пул соединений, lag репликации, глубина очередей
- [ ] Есть хотя бы 2–3 бизнес-метрики
- [ ] Для каждой метрики ясно, какое решение она помогает принять
