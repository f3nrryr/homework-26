## Проектирование высоконагруженной системы для обработки пользовательских запросов

### Цель
Самостоятельно спроектировать архитектуру системы, применив все полученные знания.

### Темы для проектной работы

1. Мессенджер с групповыми чатами — аналог Telegram
2. Сервис объявлений с доставкой — аналог Avito + доставка
3. Система управления мероприятиями — билеты на концерты
4. Сервис телеметрии — для мобильных приложений
5. Социальная сеть для профессионалов — аналог LinkedIn
6. Своя тема — согласовать с наставником

Важно: Нельзя выбирать ту же систему, что в ДЗ 2-5. Это должна быть новая система.

### Задание
1. **Требования**
    - Функциональные требования (scope проекта)
    - Нефункциональные требования (конкретные цифры)
    - Риски и ограничения
    - Backlog (что не входит в scope)

2. **Концептуальная архитектура**
    - Схема концептуальной архитектуры (C4: Context + Container)
    - Описание компонентов (1-2 предложения на каждый)
    - Обоснование архитектурного стиля
    - ADR для 2-3 ключевых решений
    - Пользователи и внешние интеграции

3. **Сайзинг**
    - Расчёт RPS, хранилища, bandwidth
    - Оценка ресурсов и стоимости

4. **Хранение данных**
    - Выбор БД с обоснованием
    - Схема шардирования
    - Стратегия кэширования

5. **Взаимодействие**
    - Схема взаимодействия сервисов
    - Выбор протоколов
    - Ключевые API (3-5 контрактов)

6. **Надёжность**
    - RTO/RPO по сервисам
    - Схема репликации
    - План DR

7. **Безопасность**
    - Аутентификация/авторизация
    - Шифрование и управление секретами

8. **Observability**
    - Ключевые метрики (10-15)
    - Алерты (5-7)

9. **Тестирование**
    - План нагрузочного тестирования
    - Chaos-эксперименты

### Формат сдачи
- Презентация
  - Презентация 10-15 слайдов (Шаблон в материалах занятия)
  - Защита перед комиссией

- Полная документация проекта
  - Документ с требованиями
  - Схема архитектуры C4 (PNG/PlantUML/Structurizr)
  - ADR для ключевых решений
  - Описание и обоснование
  - Расчёты с формулами
  - Схемы данных и взаимодействия
  - API контракты

### Критерии оценки
1. Полнота и конкретность требований (НФТ с цифрами)
2. Адекватный scope для курсового проекта
3. Обоснованный выбор архитектуры
4. Понятная схема с легендой
5. Корректные расчёты
6. Архитектура соответствует расчётам
7. Продуманное хранение данных
8. Логичное взаимодействие
9. Полнота архитектурного решения
10. Согласованность всех частей
11. Качество презентации
12. Ответы на вопросы комиссии
