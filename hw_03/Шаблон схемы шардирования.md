# Шаблон схемы шардирования (ДЗ 3)

Опора для пункта «Шардирование». Шаблон помогает структурировать решение, но
**ключ, стратегию и количество шардов выбираете вы** под свои данные и запросы.
Оценивается обоснование выбора ключа, а не сам факт «нарезал на N шардов».

---

## Шаг 0. А нужно ли шардировать?

Сначала ответьте — иначе можно усложнить на пустом месте:

- [ ] Данные/нагрузка реально не влезают в одну ноду (с репликами для чтения)?
- [ ] Вертикальное масштабирование и кэш уже исчерпаны или скоро исчерпаются?
- [ ] Есть оценка объёма/RPS, которая это подтверждает (хотя бы грубая)?

> Если нет — честно напишите «шардирование не требуется, обоснование: …». Это валидный ответ.

---

## Шаг 1. Выбор ключа шардирования

Ключ — главное решение. Хороший ключ даёт **равномерность** и **локальность запросов**.

Вопросы к кандидату в ключи:
1. **Равномерно ли распределение?** Не будет ли «горячих» значений (популярный продавец, город-миллионник)?
2. **По какому полю идёт большинство запросов?** Попадают ли они в один шард или размазываются по всем?
3. **Растёт ли множество значений** со временем (можно ли добавлять шарды)?
4. **Нужны ли запросы по диапазону** (тогда range) или только точечные (тогда hash)?
5. **Где транзакции?** Лежат ли связанные данные в одном шарде?

| Кандидат в ключи | Равномерность | Локальность запросов | Риск hotspot | Вывод |
|---|---|---|---|---|
| _напр. user_id_ | _высокая_ | _запросы по пользователю — в один шард_ | _низкий_ | _подходит_ |
| … | | | | |

## Шаг 2. Стратегия шардирования

| Стратегия | Как работает | Плюсы | Минусы / когда осторожно |
|---|---|---|---|
| **Range** (по диапазону) | ключи делятся на интервалы | запросы по диапазону, сортировка | риск hotspot на «свежих» данных |
| **Hash** | shard = hash(key) % N | равномерность | теряется диапазон; resize ломает % N |
| **Consistent hashing** | ключи и ноды на «кольце» | плавная ребалансировка | сложнее реализация |
| **Directory / lookup** | таблица «ключ → шард» | гибкость, ручной контроль | lookup-сервис = точка отказа |
| **Geo / по атрибуту** | по региону/тенанту | data locality, compliance | перекос между регионами |

## Шаг 3. Ребалансировка и рост

- [ ] Как добавляются новые шарды? (consistent hashing / virtual nodes / pre-splitting)
- [ ] Что происходит с данными при добавлении шарда — сколько ключей переедет?
- [ ] Как избегаем простоя при ребалансировке?

---

## Шаг 4. Проблемы, которые надо адресовать

- [ ] **Hotspots** — есть ли перекос, как лечим (соль к ключу, более дробный ключ)?
- [ ] **Cross-shard запросы** — какие запросы затрагивают несколько шардов, как с ними быть (scatter-gather, денормализация)?
- [ ] **Распределённые транзакции** — есть ли они, как обходимся (Saga, держать связанное в одном шарде)?
- [ ] **Вторичные индексы** — локальные на шард или глобальные?
