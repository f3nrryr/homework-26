# Таблица сравнения баз данных (ДЗ 3)

Опора для пункта «Выбор БД». Таблица напоминает свойства семейств БД — но
**выбор под каждый сервис делаете вы**, привязывая к своей модели данных и НФТ.
Главный антипаттерн, который оценивается: «PostgreSQL везде» или «модный NoSQL без причины».

---

## Семейства БД: матрица свойств

| Семейство | Примеры | Модель данных | Сильные стороны | Слабые стороны | Консистентность по умолчанию | Масштабирование |
|---|---|---|---|---|---|---|
| **Реляционные** | PostgreSQL, MySQL | таблицы, строгая схема, JOIN | транзакции ACID, целостность, гибкие запросы | вертикальный потолок, шардирование вручную | строгая | в основном вертикальное (+ реплики) |
| **Документные** | MongoDB, Couchbase | JSON-документы, гибкая схема | удобно для агрегатов, быстрая итерация | слабее с JOIN/транзакциями между документами | настраиваемая | горизонтальное (sharding) |
| **Key-Value** | Redis, DynamoDB | ключ → значение | очень быстрый доступ по ключу, простота | нет сложных запросов | от eventual до строгой | горизонтальное |
| **Wide-column** | Cassandra, ScyllaDB | строки с динамическими колонками | огромный объём записи, гео-распределение | запросы только по ключу партиции, денормализация | tunable (quorum) | горизонтальное, masterless |
| **Поисковые** | Elasticsearch, OpenSearch | инвертированный индекс | полнотекстовый поиск, аналитика, фасеты | не основное хранилище, near-real-time | eventual | горизонтальное |
| **Колоночные / OLAP** | ClickHouse, BigQuery | колоночное хранение | агрегаты по миллиардам строк | не для точечных update/мелких запросов | — | горизонтальное |
| **Time-series** | TimescaleDB, InfluxDB | метки времени | дозапись и запросы по времени, retention | узкая специализация | — | горизонтальное |
| **Графовые** | Neo4j, JanusGraph | узлы и связи | обход связей, рекомендации, графы | плохо масштабируется на huge graph | строгая (часто) | ограниченное |
| **NewSQL** | CockroachDB, YugabyteDB, Spanner | SQL + распределённость | ACID + горизонтальное масштабирование | сложность, латентность на запись | строгая (распределённая) | горизонтальное |

## Вопросы для выбора БД под сервис

Пройдите по ним **для каждого сервиса** из ДЗ 2 — ответы и приводят к выбору:

1. **Какая форма у данных?** Жёсткие связи и транзакции / самодостаточные документы / пары ключ-значение / временные ряды / граф связей?
2. **Какие запросы преобладают?** Точечный доступ по ключу / сложные JOIN и фильтры / полнотекстовый поиск / агрегаты / обход связей?
3. **Соотношение чтение/запись?** И каков их объём (из расчётов ДЗ 4, если уже есть)?
4. **Нужны ли ACID-транзакции?** Между какими сущностями? Можно ли жить с eventual consistency?
5. **Какой масштаб данных и рост?** Влезает в одну ноду или нужно горизонтальное масштабирование?
6. **Требования к латентности и доступности?** (связь с CAP — см. пункт 6 ДЗ)
7. **Что уже есть в команде/инфраструктуре?** (валидный критерий — операционная сложность стоит денег)

## Шаблон для заполнения (таблица выбора БД)

| Сервис | Что хранит | Форма данных | Ключевые запросы | Выбранная БД | Почему именно она (1–2 фразы) |
|---|---|---|---|---|---|
| _напр. orders_ | _заказы, статусы_ | _реляционная, транзакции_ | _по id, по пользователю_ | _PostgreSQL_ | _нужны ACID при оформлении_ |
| … | | | | | |

> Хороший ответ: разные сервисы — разные БД, и для каждого видно, **почему**.
> Слабый ответ: одна БД на всё «потому что знаю её» либо NoSQL «потому что хайлоад».

## Частые ловушки (самопроверка)

- [ ] Не выбрал одну БД «по умолчанию» для всех сервисов без разбора
- [ ] Не взял экзотическую БД ради «модности» — есть конкретная причина
- [ ] Учёл операционную стоимость (ещё одна СУБД = ещё одна точка эксплуатации)
- [ ] Выбор согласован с типом запросов, а не только с объёмом данных
- [ ] Где выбран NoSQL — явно сказал, чем пожертвовал (JOIN, транзакции, консистентность)
