# Таблица сравнения протоколов взаимодействия (ДЗ 2)

Шпаргалка для выбора протоколов в ДЗ 2. Используйте её как опору для пункта
«Протоколы» и для обоснования: под каждый тип взаимодействия — свой инструмент.
Главный принцип: **нет «лучшего» протокола, есть подходящий под сценарий**.

---

## Основная матрица

| Критерий | REST | gRPC | GraphQL | WebSocket | SSE | Message broker (Kafka/RabbitMQ) | MQTT |
|---|---|---|---|---|---|---|---|
| **Модель взаимодействия** | request/response | request/response + streaming | query/mutation/subscription | full-duplex, двунаправленный | server→client, однонаправленный | async pub/sub, очереди | async pub/sub |
| **Транспорт** | HTTP/1.1, HTTP/2 | HTTP/2 | HTTP (обычно поверх REST) | TCP (поверх HTTP upgrade) | HTTP | TCP (свой протокол) | TCP (поверх него), есть over-WS |
| **Формат данных** | JSON/XML (текст) | Protobuf (бинарь) | JSON по GraphQL-схеме | любой | текст (UTF-8) | любой (часто Avro/Protobuf/JSON) | бинарь/любой, компактный |
| **Типизация контракта** | слабая (OpenAPI опционально) | строгая (.proto) | строгая (SDL) | нет из коробки | нет | по схеме (Schema Registry) | нет |
| **Производительность** | средняя | высокая | средняя | высокая, low latency | высокая для потока | очень высокая (throughput) | высокая, экономичная |
| **Стриминг** | нет (только polling) | да (client/server/bi-di) | subscriptions | да | да (server→client) | да (event streaming) | да |
| **Поддержка в браузере** | нативная | через grpc-web/прокси | нативная | нативная | нативная | нет (только через шлюз) | через MQTT-over-WS |
| **Сложность внедрения** | низкая | средняя/высокая | средняя | средняя | низкая | высокая (инфраструктура) | средняя |
| **Наблюдаемость/дебаг** | простой (curl, браузер) | сложнее (бинарь) | средний | сложнее | простой | средний (нужны инструменты) | средний |

## Когда применять / минусы

| Протокол | Когда применять | Минусы / на что смотреть |
|---|---|---|
| **REST** | Публичные API, CRUD, простая интеграция, кэшируемые ресурсы | Overfetching/underfetching, версионирование, нет стриминга |
| **gRPC** | Межсервисное взаимодействие, low-latency, строгий контракт, polyglot | Бинарь (сложнее дебаг), слабая поддержка в браузере, не для публичных API |
| **GraphQL** | Агрегация данных под нужды клиента, гибкие выборки, много типов клиентов (web/mobile) | Сложность сервера, N+1, кэширование труднее, риск тяжёлых запросов |
| **WebSocket** | Real-time двусторонний обмен: чаты, презентация, совместное редактирование, статусы поездки | Держит соединение (ресурсы), сложнее масштабировать, нет авто-reconnect из коробки |
| **SSE** | Однонаправленные обновления server→client: лента событий, нотификации, прогресс | Только текст, только в одну сторону, лимит соединений на HTTP/1.1 |
| **Message broker** | Развязка сервисов, асинхронность, буфер пиков, событийность, гарантии доставки | Eventual consistency, инфраструктура и эксплуатация, порядок/дубликаты |
| **MQTT** | IoT, телеметрия, мобильные/нестабильные сети, экономия трафика и батареи | Не для request/response, нужен брокер, ограниченный размер полезной нагрузки |

## Эволюция транспорта (контекст, не выбор «вместо»)

Это не альтернативы протоколам выше, а версии HTTP, на которых они работают:

| Версия | Что даёт | Влияние на архитектуру |
|---|---|---|
| **HTTP/1.1** | Базовый, текстовый, 1 запрос на соединение (keep-alive) | Head-of-line blocking, нужно много соединений |
| **HTTP/2** | Мультиплексирование, бинарные фреймы, server push | Основа gRPC; меньше соединений, выше throughput |
| **HTTP/3 (QUIC)** | Поверх UDP, нет HOL blocking на уровне транспорта, быстрый handshake | Лучше на нестабильных/мобильных сетях, быстрее установка соединения |

## Как выбирать (вопросы к каждому ребру взаимодействия)

Не сопоставляйте сценарий с готовым ответом — пройдите по вопросам для **каждой пары
«кто с кем общается»** в вашей системе и сделайте вывод сами:

1. **Кто инициатор и кто отвечает?** Это команда (запрос-ответ) или факт о произошедшем (событие)?
2. **Нужен ли ответ синхронно?** Или вызывающий может продолжить работу, не дожидаясь?
3. **Кто потребитель — браузер, мобильный клиент или другой сервис?** Какие у него ограничения?
4. **Какой latency-бюджет на этом ребре?** Сколько миллисекунд допустимо?
5. **Нужна ли строгая типизация контракта** и кросс-языковая генерация клиентов?
6. **Есть ли поток данных?** В одну сторону (server→client) или в обе?
7. **Нужно ли пережить пики и развязать сервисы во времени?** Важны ли гарантии доставки и порядок?
8. **Какая сеть и устройство?** Стабильный ДЦ или мобильная/IoT-сеть с экономией трафика?
9. **Что умеет команда и что уже есть в инфраструктуре?** (тоже валидный критерий)

Ответив на это, сверьтесь с матрицей выше и **обоснуйте выбор словами своей системы**,
а не общими свойствами протокола.

> В реальной системе протоколы **сочетаются**: REST/gRPC для команд, брокер для событий,
> WebSocket/SSE для real-time. В ДЗ 2 ожидается именно такой осознанный микс с обоснованием.
